# Requirements Document

## Introduction

This document specifies requirements for adding QR code generation and multiple payment methods (specifically card payments) to the OnePay payment verification platform. OnePay currently supports only bank transfers via the Quickteller Dynamic Transfer API. These enhancements will enable merchants to offer customers multiple payment options and provide QR codes containing complete payment details for easy scanning and sharing.

## Glossary

- **Payment_System**: The OnePay Flask-based payment verification platform
- **Merchant**: A registered user who creates payment links
- **Customer**: An end user who receives a payment link and makes a payment
- **Payment_Link**: A time-bound, single-use URL generated by a Merchant for a specific transaction
- **QR_Code**: A machine-readable optical label containing payment information
- **Payment_Method**: A mechanism for completing payment (bank transfer or card payment)
- **Card_Payment**: Payment made using credit or debit card via Quickteller Card Payment API
- **Bank_Transfer**: Payment made via virtual account using Quickteller Dynamic Transfer API
- **Processing_Fee**: A percentage-based charge applied to card payments (1.5%)
- **Transaction**: A database record representing a payment request with associated metadata
- **Payment_Page**: The customer-facing page at /pay/<tx_ref> where payment details are displayed
- **Quickteller_API**: Interswitch payment gateway API for processing payments
- **Mock_Mode**: Testing mode that simulates API responses without real credentials

## Requirements

### Requirement 1: QR Code Generation for Payment Links

**User Story:** As a Customer, I want to scan a QR code on the payment page, so that I can quickly access payment details on my mobile device or share them with others.

#### Acceptance Criteria

1. WHEN a Customer accesses a valid Payment_Link, THE Payment_System SHALL display a QR_Code on the Payment_Page
2. THE QR_Code SHALL encode structured payment data including transaction reference, amount, currency, merchant information, and payment URL
3. WHEN a Customer scans the QR_Code, THE Payment_System SHALL decode the data and display payment details
4. THE Payment_System SHALL generate QR_Codes using a Python QR code library with error correction level M
5. THE QR_Code SHALL remain visible throughout the payment session until the Transaction expires or is verified
6. WHEN the Transaction is expired, THE Payment_System SHALL not display a QR_Code
7. WHEN the Transaction is verified, THE Payment_System SHALL display a success state without the QR_Code

### Requirement 2: Payment Method Selection by Merchant

**User Story:** As a Merchant, I want to specify which payment methods are allowed when creating a Payment_Link, so that I can control how Customers can pay me.

#### Acceptance Criteria

1. WHEN a Merchant creates a Payment_Link, THE Payment_System SHALL accept a list of allowed Payment_Methods
2. THE Payment_System SHALL support two Payment_Methods: bank_transfer and card_payment
3. WHERE no Payment_Methods are specified, THE Payment_System SHALL default to bank_transfer only for backward compatibility
4. THE Payment_System SHALL validate that at least one Payment_Method is specified
5. THE Payment_System SHALL store allowed Payment_Methods in the Transaction database record
6. WHEN a Payment_Method is not supported, THE Payment_System SHALL return a validation error with code INVALID_PAYMENT_METHOD

### Requirement 3: Payment Method Display on Payment Page

**User Story:** As a Customer, I want to see available payment options on the Payment_Page, so that I can choose my preferred payment method.

#### Acceptance Criteria

1. WHEN a Customer accesses a Payment_Page, THE Payment_System SHALL display all allowed Payment_Methods for that Transaction
2. WHERE only one Payment_Method is allowed, THE Payment_System SHALL display that method without a selection interface
3. WHERE multiple Payment_Methods are allowed, THE Payment_System SHALL display a selection interface with radio buttons or tabs
4. WHEN a Customer selects a Payment_Method, THE Payment_System SHALL display the appropriate payment instructions for that method
5. THE Payment_System SHALL display bank transfer details WHEN bank_transfer is selected
6. THE Payment_System SHALL display card payment form WHEN card_payment is selected

### Requirement 4: Card Payment Integration

**User Story:** As a Customer, I want to pay using my credit or debit card, so that I have an alternative to bank transfers.

#### Acceptance Criteria

1. WHEN card_payment is an allowed Payment_Method, THE Payment_System SHALL integrate with Quickteller Card Payment API
2. THE Payment_System SHALL collect card number, expiry date, CVV, and cardholder name from the Customer
3. WHEN a Customer submits card details, THE Payment_System SHALL validate card number format using Luhn algorithm
4. WHEN a Customer submits card details, THE Payment_System SHALL validate expiry date is in the future
5. WHEN a Customer submits card details, THE Payment_System SHALL validate CVV is 3 or 4 digits
6. THE Payment_System SHALL transmit card details to Quickteller Card Payment API over HTTPS
7. WHEN Quickteller returns response code 00, THE Payment_System SHALL mark the Transaction as verified
8. WHEN Quickteller returns a non-00 response code, THE Payment_System SHALL display an error message to the Customer
9. THE Payment_System SHALL not store raw card numbers, CVV, or full card details in the database
10. THE Payment_System SHALL store only the last 4 digits of the card number and card brand for Transaction records

### Requirement 5: Processing Fee Calculation and Display

**User Story:** As a Merchant, I want processing fees for card payments to be passed to the Customer, so that I receive the full requested amount.

#### Acceptance Criteria

1. WHEN card_payment is selected, THE Payment_System SHALL calculate a processing fee of 1.5% of the Transaction amount
2. THE Payment_System SHALL display the original amount, processing fee, and total amount to the Customer before payment
3. THE Payment_System SHALL charge the Customer the total amount (original amount plus processing fee)
4. THE Payment_System SHALL record the original amount, processing fee, and total amount in the Transaction record
5. WHEN bank_transfer is selected, THE Payment_System SHALL not apply any processing fee
6. THE Payment_System SHALL use Decimal type for all fee calculations to ensure precision
7. THE Payment_System SHALL round processing fees to 2 decimal places using ROUND_HALF_UP

### Requirement 6: Database Schema for Payment Methods

**User Story:** As a Developer, I want the database schema to support multiple payment methods and fees, so that the system can track payment method usage and fee amounts.

#### Acceptance Criteria

1. THE Payment_System SHALL add an allowed_payment_methods column to the transactions table storing a JSON array
2. THE Payment_System SHALL add a selected_payment_method column to the transactions table storing a string value
3. THE Payment_System SHALL add a processing_fee column to the transactions table storing a Decimal value
4. THE Payment_System SHALL add a total_amount column to the transactions table storing a Decimal value
5. THE Payment_System SHALL create a database migration script to add these columns
6. THE Payment_System SHALL set default values for existing Transaction records: allowed_payment_methods=['bank_transfer'], selected_payment_method='bank_transfer', processing_fee=0, total_amount=amount
7. THE Payment_System SHALL maintain backward compatibility with existing Transaction records

### Requirement 7: Card Payment Status Tracking

**User Story:** As a Merchant, I want to see which payment method was used for each Transaction, so that I can track payment method preferences and reconcile payments.

#### Acceptance Criteria

1. WHEN a Customer completes a payment, THE Payment_System SHALL record the selected_payment_method in the Transaction
2. WHEN a card payment is initiated, THE Payment_System SHALL update Transaction status to pending
3. WHEN a card payment succeeds, THE Payment_System SHALL update Transaction status to verified
4. WHEN a card payment fails, THE Payment_System SHALL update Transaction status to failed
5. THE Payment_System SHALL display the selected_payment_method in the Merchant dashboard transaction history
6. THE Payment_System SHALL include selected_payment_method in CSV export files
7. THE Payment_System SHALL include selected_payment_method in webhook payloads

### Requirement 8: Security for Card Payment Data

**User Story:** As a Security Officer, I want card payment data to be handled securely, so that Customer payment information is protected according to PCI-DSS standards.

#### Acceptance Criteria

1. THE Payment_System SHALL transmit card details to Quickteller API using TLS 1.2 or higher
2. THE Payment_System SHALL not log card numbers, CVV, or expiry dates in application logs
3. THE Payment_System SHALL not store card numbers, CVV, or expiry dates in the database
4. THE Payment_System SHALL validate and sanitize all card payment input fields
5. THE Payment_System SHALL implement rate limiting on card payment submission endpoints (5 attempts per 15 minutes per IP)
6. THE Payment_System SHALL audit log all card payment attempts with transaction reference and outcome
7. WHEN a card payment fails, THE Payment_System SHALL not expose detailed error messages that could aid attackers
8. THE Payment_System SHALL use CSRF tokens on all card payment form submissions

### Requirement 9: QR Code Data Format

**User Story:** As a Developer, I want QR codes to contain structured payment data, so that scanning applications can parse and display payment information correctly.

#### Acceptance Criteria

1. THE QR_Code SHALL encode data in JSON format
2. THE QR_Code data SHALL include fields: tx_ref, amount, currency, description, merchant_name, payment_url, expires_at
3. THE QR_Code data SHALL include virtual_account_number, virtual_bank_name, virtual_account_name WHEN bank_transfer is available
4. THE QR_Code data SHALL include allowed_payment_methods array
5. THE Payment_System SHALL validate QR_Code data size does not exceed QR code capacity (version 10, approximately 1000 characters)
6. WHEN QR_Code data exceeds capacity, THE Payment_System SHALL omit optional fields (description, merchant_name) to reduce size
7. THE Payment_System SHALL generate QR_Codes as PNG images with minimum size 200x200 pixels

### Requirement 10: Mock Mode Support for Card Payments

**User Story:** As a Developer, I want to test card payments without real API credentials, so that I can develop and test the feature locally.

#### Acceptance Criteria

1. WHEN Quickteller card payment credentials are not configured, THE Payment_System SHALL operate in Mock_Mode for card payments
2. WHEN in Mock_Mode, THE Payment_System SHALL accept any valid card number format
3. WHEN in Mock_Mode, THE Payment_System SHALL return success (response code 00) for card numbers ending in even digits
4. WHEN in Mock_Mode, THE Payment_System SHALL return failure (response code 06) for card numbers ending in odd digits
5. WHEN in Mock_Mode, THE Payment_System SHALL display a banner on the Payment_Page indicating mock mode is active
6. THE Payment_System SHALL log clearly when operating in Mock_Mode for card payments

### Requirement 11: Backward Compatibility with Existing Payment Links

**User Story:** As a Merchant, I want my existing payment links to continue working, so that Customers can still complete payments using old links.

#### Acceptance Criteria

1. WHEN a Transaction was created before the payment methods feature, THE Payment_System SHALL treat it as bank_transfer only
2. THE Payment_System SHALL display bank transfer details for legacy Transactions without showing payment method selection
3. THE Payment_System SHALL not require allowed_payment_methods field for legacy Transactions
4. THE Payment_System SHALL continue to support the existing /pay/<tx_ref> URL format
5. THE Payment_System SHALL continue to support the existing polling mechanism for bank transfer confirmation
6. THE Payment_System SHALL maintain existing webhook delivery behavior for legacy Transactions

### Requirement 12: QR Code Library Selection

**User Story:** As a Developer, I want to use a well-maintained QR code library, so that QR code generation is reliable and secure.

#### Acceptance Criteria

1. THE Payment_System SHALL use the qrcode Python library (version 7.0 or higher)
2. THE Payment_System SHALL configure QR_Codes with error correction level M (15% recovery)
3. THE Payment_System SHALL generate QR_Codes with box_size of 10 pixels and border of 4 boxes
4. THE Payment_System SHALL serve QR_Code images as base64-encoded data URLs embedded in HTML
5. THE Payment_System SHALL cache generated QR_Codes for the duration of the Transaction lifetime
6. WHEN QR code generation fails, THE Payment_System SHALL log the error and display the Payment_Page without the QR_Code

### Requirement 13: Card Payment API Configuration

**User Story:** As a System Administrator, I want to configure card payment API credentials via environment variables, so that I can deploy the system securely.

#### Acceptance Criteria

1. THE Payment_System SHALL read Quickteller card payment credentials from environment variables
2. THE Payment_System SHALL require QUICKTELLER_CARD_MERCHANT_ID environment variable for card payments
3. THE Payment_System SHALL require QUICKTELLER_CARD_API_KEY environment variable for card payments
4. THE Payment_System SHALL require QUICKTELLER_CARD_TERMINAL_ID environment variable for card payments
5. WHEN card payment credentials are not set, THE Payment_System SHALL operate in Mock_Mode
6. THE Payment_System SHALL validate card payment credentials are not placeholder values on startup in production mode
7. THE Payment_System SHALL log a warning WHEN card payment is configured but credentials are missing

### Requirement 14: Payment Method Fee Display

**User Story:** As a Customer, I want to see fee information before selecting a payment method, so that I can make an informed decision.

#### Acceptance Criteria

1. WHEN multiple Payment_Methods are available, THE Payment_System SHALL display fee information for each method
2. THE Payment_System SHALL display "No additional fees" for bank_transfer
3. THE Payment_System SHALL display "1.5% processing fee" for card_payment
4. WHEN a Customer selects card_payment, THE Payment_System SHALL display a fee breakdown showing original amount, fee amount, and total
5. THE Payment_System SHALL update the displayed total amount dynamically WHEN the Customer changes payment method selection
6. THE Payment_System SHALL display all amounts in the Transaction currency with 2 decimal places

### Requirement 15: Round-Trip Property for QR Code Data

**User Story:** As a Developer, I want to verify QR code encoding and decoding work correctly, so that scanned QR codes contain accurate payment information.

#### Acceptance Criteria

1. THE Payment_System SHALL implement a QR_Code encoder function that converts Transaction data to QR_Code image
2. THE Payment_System SHALL implement a QR_Code decoder function that extracts Transaction data from QR_Code image
3. FOR ALL valid Transactions, encoding then decoding the QR_Code SHALL produce equivalent payment data
4. THE Payment_System SHALL validate that decoded tx_ref matches the original Transaction reference
5. THE Payment_System SHALL validate that decoded amount matches the original Transaction amount
6. THE Payment_System SHALL validate that decoded payment_url is a valid URL pointing to the correct Payment_Page
